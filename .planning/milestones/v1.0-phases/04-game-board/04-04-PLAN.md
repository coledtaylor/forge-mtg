1. **Create ActionBar.tsx:**
   - Props: `{ wsRef: React.MutableRefObject<GameWebSocket | null>; className?: string }`
   - Read from store: `prompt`, `buttons`, `humanPlayerId`, `activePlayerId`
   - 44px tall bar, card background color, fixed above hand area, col-span-2
   - Layout: flex row, items-center, justify-between, px-md (16px), gap-md (16px)

   **Button mode (buttons !== null):**
   - When BUTTON_UPDATE is active, show the engine-provided button labels
   - Left side: prompt text (from the most recent prompt message, or "Priority" as default)
   - Right side: up to 3 buttons:
     - Button 1 (primary): label from `buttons.label1`, enabled per `buttons.enable1`, calls `wsRef.current?.sendButtonOk()`. Use shadcn Button with `variant="default"` (primary).
     - Button 2 (secondary): label from `buttons.label2`, enabled per `buttons.enable2`, calls `wsRef.current?.sendButtonCancel()`. Use shadcn Button with `variant="outline"`.
     - "Pass Priority" button: always visible, calls `wsRef.current?.sendButtonOk()`. Use shadcn Button with `variant="ghost"`.
   - Focus: if `buttons.focus1` is true, auto-focus button 1

   **Prompt mode (prompt !== null):**
   - Determine prompt type from `prompt.type`:
     - `PROMPT_CHOICE`: render ChoiceDialog inline with choices from `prompt.payload.choices` or `prompt.payload.options`
     - `PROMPT_CONFIRM`: show prompt message + "Yes"/"No" buttons (use `prompt.payload.yesButton` / `prompt.payload.noButton` labels if provided, else "Yes"/"No"). Yes calls `wsRef.current?.sendConfirmResponse(prompt.inputId, true)`, No calls `sendConfirmResponse(prompt.inputId, false)`.
     - `PROMPT_AMOUNT`: show prompt message + number input with min/max from payload + "OK" button. OK calls `wsRef.current?.sendAmountResponse(prompt.inputId, amount)`.
   - After sending any response, call `store.setPrompt(null)` to clear the prompt

   **Idle state (no prompt, no buttons):**
   - Show "Waiting for opponent..." in muted-foreground text

   **Active prompt styling:**
   - When prompt or buttons are active: add `border-t-2 border-primary` to the bar (accent glow per UI-SPEC)
   - When idle: no border-top

2. **Create ChoiceDialog.tsx:**
   - Props: `{ prompt: PromptState; wsRef: React.MutableRefObject<GameWebSocket | null> }`
   - Read choices from `prompt.payload.choices` or `prompt.payload.options` (whichever is present)
   - Layout: inline within ActionBar (non-modal per user decision)
   - If choices array is short (<=5): render as a row of shadcn Buttons, one per choice
   - If choices array is long (>5): render as a scrollable list with items, each clickable
   - Selection: single-select by default. If `prompt.payload.max > 1`, support multi-select with checkboxes and a "Confirm" button
   - On select (single): immediately send `wsRef.current?.sendChoiceResponse(prompt.inputId, [selectedIndex])` and clear prompt
   - On confirm (multi): send `wsRef.current?.sendChoiceResponse(prompt.inputId, selectedIndices)` and clear prompt
   - Show prompt message above choices: `prompt.payload.message` in 14px regular
   - If `prompt.payload.isOptional` is true, show a "Skip" button that sends empty selection

3. **Create GameOverScreen.tsx:**
   - Props: `{ onReturnToLobby: () => void }`
   - Read from store: `gameOver` state
   - Return null if `gameOver` is null
   - Full-screen overlay: fixed inset-0, `bg-black/70` backdrop (board visible but dimmed)
   - Centered container: card background, 400px wide, rounded-lg, shadow-2xl, p-8, flex flex-col items-center gap-4
   - Result heading: 20px semibold
     - If gameOver.winner matches human player name: "You Won!" in `text-green-400`
     - If gameOver.winner does not match: "You Lost" in `text-red-400`
     - If draw: "Draw" in `text-muted-foreground`
   - Subtext: 14px regular muted-foreground, show `gameOver.message`
   - Primary action: shadcn Button `variant="default"`, text "Return to Lobby", calls `onReturnToLobby`
   - Secondary action: shadcn Button `variant="outline"`, text "View Board", dismisses the overlay (sets local `dismissed` state to true, returns null when dismissed)
  </action>

1. **Create CombatOverlay.tsx:**
   - Props: `{ className?: string }`
   - Read from store: `combat`, `cards`, `humanPlayerId`
   - Return null if combat is null or combat.attackers is empty
   - SVG overlay that covers the battlefield area (absolute positioned over rows 2-4 of the grid)
   - `pointer-events-none` so clicks pass through to cards beneath (except for interactive blocker assignment)

   **Attacker visualization:**
   - For each attacker in combat.attackers: look up card position on the DOM using `data-card-id` attributes on GameCard elements
   - Draw a subtle red glow indicator on attacker positions (or rely on GameCard's highlightMode="attacker" -- see wiring below)

   **Blocker assignment arrows:**
   - For each attacker that has blockerCardIds: draw SVG lines/arrows from each blocker's position to the attacker's position
   - Arrow style: `stroke="rgb(250, 204, 21)"` (yellow-400), `stroke-width="2"`, with arrowhead marker
   - Use `document.querySelector(`[data-card-id="${cardId}"]`)?.getBoundingClientRect()` to get positions relative to the overlay
   - Recalculate positions on combat state change using useEffect

   **Blocker assignment interaction (during COMBAT_DECLARE_BLOCKERS phase):**
   - Managed at GameBoard level: when phase is "COMBAT_DECLARE_BLOCKERS" and it's the human player's turn to block:
     - Track `selectedBlocker: number | null` state at GameBoard level
     - First click on an eligible blocker card: set selectedBlocker
     - Second click on an attacker card: assign the block (add to local blocker assignments)
     - Click assigned blocker again: unassign
     - These assignments are tracked locally until OK is pressed, then sent via prompt response
   - CombatOverlay reads both the server's combat state AND the local pending assignments to draw arrows

   **Targeting mode wiring:**
   - Read `prompt` from store. When prompt is a PROMPT_CHOICE that involves card selection (detect by checking if prompt.payload.cardId is set or prompt.payload.message contains "target" or "choose"):
     - Set a `targetingMode` local state to true
     - Pass `highlightMode` to BattlefieldZone's cards:
       - Valid targets (cards matching the prompt criteria): `'valid-target'`
       - Invalid cards: `'invalid'`
     - On card click during targeting: send choice response with the selected card's index
     - Clear targeting mode after response
   - For a simpler v1 approach: when any PROMPT_CHOICE is active, all battlefield cards are clickable. The engine validates on its end. Show `'valid-target'` on all cards during a PROMPT_CHOICE, and let engine reject invalid choices.

   **Attacker declaration wiring:**
   - When phase is "COMBAT_DECLARE_ATTACKERS" and buttons are active:
     - Track `declaredAttackers: Set<number>` local state
     - HandZone cards: not interactive for combat
     - Player's battlefield creatures: highlight eligible (untapped creatures, no summoning sickness -- simplify to just untapped non-land cards for v1) with `'valid-target'`
     - Click toggles attacker status: add/remove from declaredAttackers set, GameCard gets `'attacker'` highlightMode when in set
     - OK button in ActionBar sends the attacker IDs (not needed for v1 -- the engine manages combat via its own prompts. The client just responds to BUTTON_OK/CANCEL for attack declaration confirmation)

   **Playable card indicators:**
   - When buttons are active (it's the human player's priority), pass `isPlayable={true}` to HandCards for all cards (simplified -- engine will reject unplayable casts). More precise: only when buttons.enable1 is true.

   **Add `data-card-id` attribute** to GameCard's root div: `data-card-id={card.id}` -- needed for CombatOverlay arrow positioning.
